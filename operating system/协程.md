# 1. 基本信息

为什么要有协程?
**同步编程, 异步性能?**

![[sync_and_async.png]]
同步是串行, 发一个请求收到一个回包后才会发下一次, 异步是一次性把for循环走完, 是并行, 对方返回后有对应的callback函数处理相关逻辑。对于CS交互而言同步是这样的, 对于服务器而言, 可能是单线程的, 处理完一个请求才能处理下一个, 这也是同步的


# 2. 协程实现方式

一般情况下实现异步编程时会存在一个callback函数, 当send信息给对方后, 对方有返回可读的数据时执行callback
```c
// callback是FD有数据时调用
func callback() {
	recv();
}

func () {
	send(fd, callback);
}
```

假如说现在打开一个网页, 有三个请求需要同时发送到服务器, 那么多半会进行一个协程的切换操作
```c
1.
func () {
	send(fd);
	swich(1 ,2); // 让出执行流?
	recv();
}

2. 
func () {
	send(fd);
	swich(2 ,3);
	recv();
}

3. 
func () {
	send(fd);
	swich(3 ,1);
	recv();
}
```

大概可以总结为这种方式的写法:
```c
func () {
	async_send();
	async_recv();
}

func async_send() {
	//先判断是否可写, 不可写就切换
	if (poll(fd, 0) ==1) {
		switch();
	}
}

func async_send() {
	//先判断是否可读, 不可读就切换
	if (poll(fd, 0) ==1) {
		switch();
	}
}
```

switch本质是函数之间的跳转, 某个fd对应send/recv函数无法执行时, 通过switch跳转到另外一个fd的send/recv函数, 实现函数之间的跳转有三种方式

- setjmp/longjmp
- ucontext
- asm(汇编)
---
## 2.1 **setjmp/longjmp**

`setjmp` + `longjmp` 本质就是**手动保存 / 恢复函数调用栈上下文，实现跨函数跳转、切换执行流**，是**用户态最原始的 “协程 / 上下文切换” 工具**。
```c
#include <setjmp.h> 

// 保存当前上下文（寄存器、栈指针、程序计数器等）到 jmp_buf
int setjmp(jmp_buf env); 
// 恢复 env 里保存的上下文，跳回 setjmp 处 
void longjmp(jmp_buf env, int val);
```
- 第一次：调用时，保存上下文，返回 `0`
- 第二次：被 `longjmp` 跳回来时，返回你传入的 `val`（不能是 0）
```c
#include <stdio.h>
#include <setjmp.h>

jmp_buf env; //上下文环境

void func(int arg) {
    printf("func: %d\n", arg);
    longjmp(env, ++arg);
}

int main() {
    int ret = setjmp(env); // 第一次调用返回0, 第二次调用返回1, ......
    if (ret == 0) {
        func(ret);
    } else if (ret == 1) {
        func(ret);
    } else if (ret == 2) {
        func(ret);
    } else if (ret == 3) {
        func(ret);
    }
    return 0;
}
```

> 那么之前的async_send和async_recv函数, 在执行时会先调用一次setjmp保留一下上下文, 然后如果读或写没就绪就直接调用longjmp去到其他的函数, 其他的函数亦是如此, 所以说每一个fd都会有一个jmp_buf的变量, 通过这个变量决定跳转到哪一个FD的执行流, 用它来实现协程时, 每一个协议都会有一个jmp_buf

set_jmp在多线程中间跳转的时候, 由于线程之间的栈不一样, 所以不利于这种情况

## 2.2 ucontext

setjmp没有独立栈，不能做真正的协程, **ucontext 自带独立栈空间**，可以创建无数个 “执行流”, 每个执行流 = 一个**用户态轻量级线程 = 协程**
 
```c
#include <ucontext.h> 

// 1. 获取当前上下文 
int getcontext(ucontext_t *ucp); 
// 2. 设置上下文（直接跳过去执行） 
int setcontext(const ucontext_t *ucp); 
// 3. 保存当前上下文，然后跳转到另一个上下文 
int swapcontext(ucontext_t *oucp, const ucontext_t *ucp); 
// 4. 构造一个上下文（指定栈、指定入口函数） 
void makecontext(ucontext_t *ucp, void (*func)(), int argc, ...);
```

```c
#include <stdio.h>
#include <ucontext.h>
#include <unistd.h>

ucontext_t ctx_main;  // 主上下文
ucontext_t ctx1;      // 协程1
ucontext_t ctx2;      // 协程2

char stack1[1024*64]; // 协程1的独立栈
char stack2[1024*64]; // 协程2的独立栈

void func1() {
    printf("func1 运行\n");
    sleep(1);
    // 切回 main
    swapcontext(&ctx1, &ctx_main);
    printf("func1 再次运行\n");
    swapcontext(&ctx1, &ctx_main);
}

void func2() {
    printf("func2 运行\n");
    sleep(1);
    swapcontext(&ctx2, &ctx_main);
    printf("func2 再次运行\n");
    swapcontext(&ctx2, &ctx_main);
}

int main() {
    // 初始化协程1
    getcontext(&ctx1);
    ctx1.uc_stack.ss_sp = stack1;
    ctx1.uc_stack.ss_size = sizeof(stack1);
    ctx1.uc_link = &ctx_main;
    makecontext(&ctx1, func1, 0);
    // 初始化协程2
    getcontext(&ctx2);
    ctx2.uc_stack.ss_sp = stack2;
    ctx2.uc_stack.ss_size = sizeof(stack2);
    ctx2.uc_link = &ctx_main;
    makecontext(&ctx2, func2, 0);
    printf("切到协程1\n");
    swapcontext(&ctx_main, &ctx1);
    printf("切到协程2\n");
    swapcontext(&ctx_main, &ctx2);
    printf("切回协程1\n");
    swapcontext(&ctx_main, &ctx1);
    printf("切回协程2\n");
    swapcontext(&ctx_main, &ctx2);
    return 0;
}
```

这里一个func就可以理解为一个协程(coroutine), 这里可以回到最开始提到的三次网络请求, 这里是使用协程做一个简单的网络IO的代码 [2.3.1-coroutine/ucontext-backup.c · 杭电码农-NEO/Linux cloud_server code_repository - 码云 - 开源中国](https://gitee.com/NEO_kou/linux-cloud_server-code_repository/blob/master/2.3.1-coroutine/ucontext-backup.c) , 一般情况下不是协程之间直接互相切换, 而是从协程切到调度器后, 由调度器决定切到哪一个协程
![[协程切换和调度.png]]
协程是依托于线程存在的, 线程是协程运行的容器, 若整个服务器只有一个线程, 那么同一时间内, 只能有一个协程在跑
